# 变更管理协议（Change Management Protocol）

> **版本**：v1.0 | **继承自**：原 SGPA §变更管理协议
> **适用范围**：SPM / SGA / SGE 全部 Skill

---

## 变更分级

当已进入后续阶段后 USER 提出需求变更时，按以下分级处理：

| 级别 | 定义 | 影响范围 | 处理方式 |
|------|------|---------|---------|
| **微调** | 不影响 interface / 架构，仅改参数或文案 | 单个文件 | 直接修改，记录在变更日志 |
| **中度** | 影响 1-2 个 Step 的产出（如新增字段、调整 Story） | 2-3 个文件 | 回退到受影响的 Step/Gate 重做 |
| **重大** | 影响核心体验定义或架构分层 | ≥4 个文件 | 回退到 SPM Step 1 重新评估 |

---

## 变更日志格式

每次变更必须在对应文档（PRD.md / TDD.md / analysis.md）中追加变更日志：

```markdown
## 变更日志

| 日期 | 级别 | 变更内容 | 影响范围 | 回退到 | 批准人 |
|------|------|----------|----------|--------|--------|
| YYYY-MM-DD | 中度 | 新增XX字段 | TDD §2 + Story #3 | SGA Step 2 | USER |
```

## 变更流程（按 Trinity Pipeline）

### SPM 阶段变更
1. USER 提出变更 → SPM 评估变更级别
2. 微调：直接修改 PRD → 记录日志
3. 中度/重大：回退到受影响的 Step → 重新执行 → 重新走 Party Review
4. PRD 签章需要重新获取

### SGA 阶段变更
1. 变更影响 PRD 内容 → 必须先回退到 SPM 修改 PRD → 重新签章
2. 仅影响架构决策 → SGA 内处理 → 更新 ADR → 重新走 Party Review
3. TDD 签章需要重新获取

### SGE 阶段变更
1. 发现 TDD 无法实现 → 触发 **BLOCKER-REPORT**
2. BLOCKER-REPORT 内容：[问题描述] + [影响范围] + [建议修改方案]
3. 打回 SGA 修改 TDD → SGA 重新签章后 SGE 继续

---

## 回滚方案

- 编码实施前，必须确保 Git 分支策略允许快速回滚（独立 feature 分支）
- 如新系统上线后发现破坏核心循环：`git revert` → 重新进入 SPM 评估
- 变更管理不是「阻止变更」，而是「让变更的代价透明可追溯」
