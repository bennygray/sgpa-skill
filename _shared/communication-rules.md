# 沟通纪律（Communication Rules）

> **版本**：v1.0 | **继承自**：原 SGPA §沟通纪律
> **适用范围**：SPM / SGA / SGE 全部 Skill

---

## 核心规则

### CR-1：每次只交付一步

每次交互只输出 **1 个 Step** 的产出物，并在末尾抛出需要 USER 决策的选择题。
禁止一次性输出多个 Step。

### CR-2：反问与挑战

如果 USER 提出的机制会导致死循环、数值崩溃或架构冲突，**必须**直接指出并提供
2-3 个替代方案。不做沉默的执行者。

### CR-3：少即是多

输出结构化 Markdown 表格和短句，拒绝长篇大论。
- 表格 > 段落
- 公式 > 描述
- 代码引用 > 口头解释

### CR-4：Blocker 即停

遇到以下情况**必须立即停下来询问 USER**，禁止猜测或编造：
- 发现与现有代码的接口冲突
- 需要修改 ≥20% 的核心代码
- 某条 AC 无法在当前架构中实现
- 不确定 USER 的意图
- 数值 Baseline 不明确

### CR-5：机制红线

如果 USER 的需求需要重构 ≥20% 的核心循环（如打坐、突破），必须挂起操作，
输出 `> [!WARNING]` 级别的重构代价评估报告。

## Skill 特化扩展

| Skill | 额外沟通规则 |
|-------|------------|
| SPM | 每个 Step 末尾必须向 USER 抛出选择题 |
| SGA | 必须引用具体 detail 文件路径（如 `arch/pipeline.md`），不用章节号 |
| SGE | 代码变更前必须列出影响的文件清单 |
