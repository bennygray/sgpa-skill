## 通用反合理化

| 你的想法 | 现实 |
|---------|------|
| "这个需求太简单，不用走完整流程" | 简单需求也有数值漏洞和架构隐患。流程对简单需求更快。 |
| "先写代码，文档后面补" | AGENTS.md §四 明确禁止。先文档 → 确认 → 编码。 |
| "只改了一点点，不用跑验证脚本" | AGENTS.md §3.7 明确要求。没有验证脚本不合并。 |
| "这一步还不够完美，再打磨一下" | 够用即可。Review 通过 + USER 确认 = 可以进入下一步。不要过度设计。 |
| "技术方案看起来靠谱，不用 PoC" | 上次 node-llama-cpp 崩了就是教训。关键技术必须 PoC 验证。 |

## SPM 专属反合理化

| 你的想法 | 现实 |
|---------|------|
| "跳过价值锚定，需求很明确" | 没有锚定的需求会范围蔓延。5-Why 对确定的需求更快。 |
| "Phase I 和 II 可以同时进行" | Contract Gate 禁止。PRD 未签章不得进入架构设计。 |
| "Pre-Mortem 太形式化了" | 你想不到的失败原因恰好是最危险的。做。 |
| "假设审计太啰嗦，明显的不用写" | 越"明显"的假设越容易被忽略。写。 |

## SGA 专属反合理化

| 你的想法 | 现实 |
|---------|------|
| "不用画依赖矩阵，改动很小" | "很小的改动"导致 3 个系统回归的案例数不胜数。 |
| "ADR 太重了，口头说说就行" | 没有 ADR 的决策 6 个月后没人记得为什么选了 A 方案。 |
| "Pipeline 挂载顺序无所谓" | idle-engine 的 8 步执行顺序有因果依赖。顺序错了 = 数值错了。 |

## SGE 专属反合理化

| 你的想法 | 现实 |
|---------|------|
| "TDD 的 interface 不太对，改一下" | 🔴 硬禁令。SGE 不得擅改 interface。打回 SGA。 |
| "回归脚本跑不过，应该是脚本的问题" | 先假设是代码的问题。修代码。脚本确认有 Bug 再改脚本。 |
| "这个边界条件不会有人触发" | R6 找茬QA 就是干这个的。处理。 |
