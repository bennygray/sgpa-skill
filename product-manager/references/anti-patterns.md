# 反模式清单 — SPM 产品层（Anti-Patterns）

> **适用**：/SPM 产出物自检
> **架构反模式已迁移至**：`architect/references/arch-anti-patterns.md`

## 一、User Story 反模式

| 反模式 | 示例 | 正确做法 |
|--------|------|---------| 
| AC 使用模糊动词 | "正确处理错误" | "显示红色提示'灵气不足'" |
| AC 缺少 Given | "当点击按钮，显示结果" | "Given 弟子在采集，When Tick 触发…" |
| Story 过大 | "实现整个弟子系统" | 拆分为独立的子 Story |
| 缺少边界 AC | 只有正常路径 | 增加异常/边界 AC |
| 依赖不明确 | "需要其他系统支持" | "依赖 Story #3（弟子状态机）" |

## 二、数值设计反模式

| 反模式 | 示例 | 正确做法 |
|--------|------|---------|
| 无公式凭感觉 | "产出大概 10-20 灵气" | "产出 = base × (1 + 弟子等级 × 0.1)" |
| 无 Sink 兜底 | "弟子无限产出灵草" | 设计灵草消耗（炼丹）和保质期上限 |
| 编造 Baseline | "假设基础灵气为 100" | 查询现有 `shared/formulas/` 获取真实值 |

## 三、流程控制反模式

| 反模式 | 示例 | 正确做法 |
|--------|------|---------|
| 跳过 Gate 门禁 | "PRD 差不多了，先做架构" | GATE 要求 USER 签章后才能进入 SGA |
| 验证结论用模糊词 | "应该没问题" | 必须基于实际运行结果陈述事实 |
| 跳过 Party Review | "内容不多，不用审查" | 每个 Gate 必须执行 Party Review |
| 未问就猜 | "假设用户的意思是…" | 遇到歧义立即停下来问 USER |
| 一次性输出多步 | 把 Step 1-3 合成一条消息 | 每次只输出 1 个 Step |
| 跳过 PoC 验证 | "这个库很成熟，直接用" | 未经验证的技术方案必须走 Step 1.5 PoC |

## 四、AI 集成反模式

| 反模式 | 示例 | 正确做法 |
|--------|------|---------|
| AI 决定数值 | 让 AI 模型决定伤害、收益等关键数值 | AI 只负责表现层，数值由确定性引擎计算 |
| 无降级方案 | AI 服务不可用时游戏卡死 | 设计 fallback 路径 |
| 上下文无边界 | 把全量游戏状态丢给 AI | 定义最小必要上下文 |
| 跳过 PoC 直接集成 | "这个模型很成熟" | 关键技术必须 Step 1.5 PoC |
| AI 输出未校验 | 直接展示模型原始输出 | 做格式校验和内容过滤 |
